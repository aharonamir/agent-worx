## What to Build

Parses the expert's natural-language answer into structured entities and relations,
checks for contradictions against the existing graph, and detects ambiguity. This is the
component that decides whether an answer is safe to write (status=`write`), conflicts
with existing knowledge (status=`conflict`), or needs clarification (status=`follow_up`).

```python
# src/apprentice/answer_processor.py

import json
from anthropic import Anthropic
from src.core.models import (
    AnswerProcessorInput, AnswerProcessorResult,
    ExtractedEntity, ExtractedRelation, ConflictDetail, KnowledgeGap
)
from src.infra.kuzu_client import execute

_client = Anthropic()

EXTRACTION_PROMPT = """Extract structured knowledge from this expert answer.

QUESTION ASKED: "{question}"
GAP BEING RESOLVED: "{concept_label}" (type: {gap_type}, namespace: {namespace})
EXPERT ANSWER: "{answer}"

Extract:
1. ENTITIES — distinct concepts mentioned (new or referencing existing ones)
2. RELATIONS — relationships between entities, in the form (source, relation_type, target)
3. CONFIDENCE — your confidence (0.0-1.0) that this extraction accurately represents
   what the expert said. Use 0.9+ only for direct, unambiguous statements.
4. AMBIGUITY — true if the answer is too vague, off-topic, or unclear to extract
   meaningful structured knowledge.

Respond ONLY with JSON in this exact format, no markdown fences:
{{
  "entities": [{{"label": "...", "entity_type": "...", "confidence": 0.0}}],
  "relations": [{{"source": "...", "target": "...", "relation_type": "...", "confidence": 0.0}}],
  "overall_confidence": 0.0,
  "is_ambiguous": false
}}
"""

CONTRADICTION_CHECK_QUERY = """
MATCH (a:Concept)-[r:RELATES_TO]->(b:Concept)
WHERE a.label = $source_label AND r.relation <> $relation_type
RETURN a.label AS existing_source, b.label AS existing_target,
       r.relation AS existing_relation, r.confidence AS existing_confidence
"""


def process_answer(payload: AnswerProcessorInput) -> AnswerProcessorResult:
    """
    Three-stage pipeline:
      1. Extract entities/relations via LLM
      2. If is_ambiguous or overall_confidence < 0.5 -> status=follow_up
      3. Check extracted relations against existing graph for contradictions
         -> status=conflict if found, else status=write
    """
    extraction = _extract(payload)

    if extraction["is_ambiguous"] or extraction["overall_confidence"] < 0.5:
        return AnswerProcessorResult(
            status="follow_up",
            confidence=extraction["overall_confidence"],
            follow_up_question=None,  # filled by T1.3's generate_followup, called by route handler
        )

    entities = [ExtractedEntity(**e) for e in extraction["entities"]]
    relations = [ExtractedRelation(**r) for r in extraction["relations"]]

    conflict = _check_contradictions(payload.agent_type_id, relations)
    if conflict is not None:
        return AnswerProcessorResult(
            status="conflict",
            entities=entities,
            relations=relations,
            confidence=extraction["overall_confidence"],
            conflict=conflict,
        )

    return AnswerProcessorResult(
        status="write",
        entities=entities,
        relations=relations,
        confidence=extraction["overall_confidence"],
    )


def _extract(payload: AnswerProcessorInput) -> dict:
    prompt = EXTRACTION_PROMPT.format(
        question=payload.question,
        concept_label=payload.gap_context.concept_label,
        gap_type=payload.gap_context.gap_type,
        namespace=payload.gap_context.namespace,
        answer=payload.answer,
    )

    response = _client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=600,
        messages=[{"role": "user", "content": prompt}],
    )

    text = response.content[0].text.strip()
    # Defensive: strip markdown fences if the model adds them despite instructions
    if text.startswith("```"):
        text = text.split("```")[1]
        if text.startswith("json"):
            text = text[4:]
    return json.loads(text.strip())


def _check_contradictions(agent_type: str, relations: list[ExtractedRelation]) -> ConflictDetail | None:
    """
    For each newly-extracted relation, check if the graph already has a DIFFERENT
    relation_type between the same source concept and a different target, at
    confidence above 0.5. If found, that's a contradiction — return the first one.

    Example: graph has (cheeseburger)-[excludes]->(pickles) at confidence 0.85.
    New answer implies (cheeseburger)-[includes]->(pickles). This is a contradiction.
    """
    for rel in relations:
        df = execute(agent_type, CONTRADICTION_CHECK_QUERY, {
            "source_label": rel.source,
            "relation_type": rel.relation_type,
        }).get_as_df()

        for _, row in df.iterrows():
            if row["existing_target"] == rel.target and row["existing_confidence"] > 0.5:
                # Same source+target pair, different relation type, existing confidence is meaningful
                return ConflictDetail(
                    existing_source=row["existing_source"],
                    existing_target=row["existing_target"],
                    existing_relation=row["existing_relation"],
                    existing_confidence=float(row["existing_confidence"]),
                    conflict_description=(
                        f"New answer implies '{rel.source}' {rel.relation_type} '{rel.target}', "
                        f"but the graph already has '{row['existing_source']}' {row['existing_relation']} "
                        f"'{row['existing_target']}' at confidence {row['existing_confidence']:.2f}."
                    ),
                )
    return None
```

### API Route

```python
# Add to src/api/routes/apprentice.py

from src.apprentice.answer_processor import process_answer
from src.apprentice.question_generator import generate_followup
from src.apprentice.graph_writer import write_to_graph          # T1.5
from src.apprentice.readiness_scorer import compute_and_store     # T1.6
from src.core.models import AnswerProcessorInput


@router.post("/{agent_type_id}/qa/answer")
async def submit_answer(agent_type_id: str, body: dict):
    agent = await get_agent_type_by_name(agent_type_id)

    if agent.phase != Phase.APPRENTICE:
        raise HTTPException(status_code=409, detail=f"Agent is in {agent.phase.value} phase, not apprentice")

    if not body.get("answer", "").strip():
        raise HTTPException(status_code=400, detail="answer must not be empty")

    payload = AnswerProcessorInput(
        agent_type_id=agent.name,
        question=body["question"],
        answer=body["answer"],
        gap_context=body["gap_context"],
    )
    result = process_answer(payload)

    response = {"status": result.status, "processor_result": result}

    if result.status == "write":
        write_to_graph(agent.name, result)                          # T1.5
        response["readiness_score"] = compute_and_store(agent.name)  # T1.6

    elif result.status == "follow_up":
        followup = generate_followup(
            agent.name, agent.goal, agent.task_boundary,
            original_question=payload.question,
            ambiguous_answer=payload.answer,
            targeting_gap=payload.gap_context,
        )
        result.follow_up_question = followup.question
        response["processor_result"] = result

    # status == "conflict": no graph write, conflict surfaced to expert as-is

    return response
```

## Acceptance Criteria

1. A clear, unambiguous answer (e.g. "Cheeseburgers always include cheddar cheese") returns `status="write"` with `confidence >= 0.5` and at least one extracted relation
2. An ambiguous answer (e.g. "I'm not sure, maybe sometimes") returns `status="follow_up"`
3. An answer that contradicts an existing high-confidence edge returns `status="conflict"` with a populated `ConflictDetail` describing both the new and existing relation
4. On `status="write"`, the API route calls `write_to_graph()` (T1.5) and `compute_and_store()` (T1.6) — verified via mock call assertions
5. On `status="follow_up"`, the API route calls `generate_followup()` (T1.3) and populates `follow_up_question`
6. On `status="conflict"`, no graph write occurs (verified: Kuzu node/edge count unchanged before/after)
7. `POST /qa/answer` with empty `answer` string returns `400`

## Notes

- **Why `overall_confidence < 0.5` triggers follow_up even when `is_ambiguous=false`:**
  the LLM's `is_ambiguous` flag is a soft signal — it can miss cases where the extraction
  technically parsed but the underlying confidence is low (e.g. the expert hedged heavily).
  The numeric threshold is the hard backstop.
- **Contradiction detection is intentionally conservative (existing_confidence > 0.5).**
  Low-confidence existing edges (the kind T1.2's gap detector specifically targets for
  re-asking) should NOT block new writes — that would create a deadlock where a
  low-confidence edge can never be corrected. Only relations the graph is already
  fairly sure about can trigger a conflict.
- **The conflict resolution loop is NOT in this task.** When `status="conflict"`, the
  result is returned to the expert (via the API response) for them to resolve in a
  subsequent turn — e.g. by clarifying which relation is correct. That resolution flow
  is part of the Q&A session UI, not this processor. This processor's job ends at
  *detecting and describing* the conflict.
- **JSON parsing defensiveness:** despite the prompt instruction "no markdown fences",
  Claude models sometimes wrap JSON in ` ```json ... ``` ` blocks. The `_extract` function
  strips these defensively rather than relying on prompt compliance alone.
- **`relation_type` vocabulary is intentionally open** (free-text string from the LLM,
  not an enum) — domain-specific relations like "includes", "excludes", "triggers",
  "requires", "modifier_of" will vary per agent type and cannot be enumerated upfront.
  The contradiction check works on *any* pair of differing relation types between the
  same source/target, regardless of vocabulary.
