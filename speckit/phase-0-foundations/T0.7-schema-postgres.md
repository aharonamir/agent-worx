## What to Build

Migration `002` adds the five remaining tables that `agent_types` (from migration `001`)
did not include: `delta_entries`, `team_compositions`, `cert_events`, `shadow_reviews`,
`escalation_events`. These are the tables every Phase 1-3 task reads from and writes to.

    op.execute("DROP TABLE IF EXISTS shadow_reviews")
    op.execute("DROP TABLE IF EXISTS cert_events")
    op.execute("DROP TABLE IF EXISTS team_compositions")

### Query Helper Module

```python
# src/infra/postgres_client.py (additions)

import json

from src.core.models import DeltaEntry, CertEvent, ShadowReview, EscalationEvent
from src.core.enums import DeltaStatus


async def insert_delta_entry(entry: DeltaEntry) -> None:
    pool = get_pool()
    async with pool.acquire() as conn:
        await conn.execute("""
            INSERT INTO delta_entries
                (id, agent_type, agent_instance_id, status, observation_type,
                 edge_data, triggered_by_task_id, confidence, created_at)
            VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
        """, entry.id, entry.agent_type, entry.agent_instance_id, entry.status.value,
             entry.observation_type.value, json.dumps(entry.edge_data), entry.triggered_by_task_id,
             entry.confidence, entry.created_at)


async def get_quarantine_entries(agent_type: str) -> list[dict]:
    pool = get_pool()
    async with pool.acquire() as conn:
        rows = await conn.fetch("""
            SELECT * FROM delta_entries
            WHERE agent_type = $1 AND status = $2
            ORDER BY created_at ASC
        """, agent_type, DeltaStatus.QUARANTINE.value)
    return [dict(r) for r in rows]


async def insert_cert_event(event: CertEvent) -> None:
    pool = get_pool()
    async with pool.acquire() as conn:
        await conn.execute("""
            INSERT INTO cert_events
                (id, agent_type, gate, decision, expert_id, readiness_score, notes, created_at)
            VALUES ($1, $2, $3, $4, $5, $6, $7, $8)
        """, event.id, event.agent_type, event.gate.value, event.decision.value,
             event.expert_id, event.readiness_score, event.notes, event.created_at)


async def insert_escalation_event(event: EscalationEvent) -> None:
    pool = get_pool()
    async with pool.acquire() as conn:
        await conn.execute("""
            INSERT INTO escalation_events
                (id, task_id, agent_type, instance_id, reason, task_summary,
                 scope_similarity_score, routed_to, created_at)
            VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
        """, event.id, event.task_id, event.agent_type, event.instance_id,
             event.reason, event.task_summary, event.scope_similarity_score,
             event.routed_to, event.created_at)
```

## Acceptance Criteria

1. `alembic upgrade head` applies both `001_initial_schema` and `002_cert_events` without error
2. All five tables exist: `delta_entries`, `team_compositions`, `cert_events`, `shadow_reviews`, `escalation_events`
3. CHECK constraints reject invalid enum values — e.g. inserting `status='bogus'` into `delta_entries` raises an error
4. `UNIQUE(agent_type, version)` on `team_compositions` rejects a duplicate version for the same agent type
5. `insert_delta_entry()` + `get_quarantine_entries()` round-trip returns the inserted entry with `status='quarantine'`
6. `alembic downgrade -1` cleanly drops all five tables in dependency-safe order

## Notes

- **CHECK constraints enforce the enums from `shared/DATA_SCHEMAS.md` at the database level.**
  This is intentional defense-in-depth — even if application code has a bug that bypasses
  Pydantic validation, Postgres rejects invalid enum values.
- **`team_compositions` UNIQUE(agent_type, version)`** is what makes Invariant I6 (immutability)
  enforceable at the DB level — you cannot accidentally overwrite version 1 with new data;
  you must insert version 2.
- **Downgrade order matters** even though there are no foreign keys between these five tables —
  Alembic applies `downgrade()` statements in the order written, so list drops in roughly
  reverse-dependency order as a matter of good practice (none of these tables reference each
  other via FK, but `agent_types` from migration 001 is conceptually a parent to all of them
  via the `agent_type` varchar column — a soft reference, not enforced).
- **`scope_similarity_score` is nullable** on `escalation_events` because not every escalation
  reason involves a similarity check — `max_iterations` and `action_prohibited` escalations
  have no similarity score to report.
