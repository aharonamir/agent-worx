## What to Build

Every Redis key in the framework follows one of the patterns below. Rather than constructing
keys with ad-hoc f-strings scattered across the codebase (which leads to typos and drift),
all key construction goes through `src/core/redis_keys.py`. Every other task (T1.1, T1.6,
T3.6, etc.) imports these functions instead of hardcoding key strings.

### Key Patterns

| Pattern | Example | Set by | Read by |
|---|---|---|---|
| `agent:{name}:phase` | `agent:burger-order:phase` | T1.1 (init), T1.11/T2.8 (cert gates) | All phases — routing decisions |
| `agent:{name}:learning_rate` | `agent:burger-order:learning_rate` | T1.1, T1.11, T2.8 | T2.x, T3.x delta thresholds |
| `agent:{name}:readiness_score` | `agent:burger-order:readiness_score` | T1.6 | T1.7 (proposal trigger), T1.11 (cert gate) |
| `agent:{name}:proposal_ready` | `agent:burger-order:proposal_ready` | T1.6 | T1.7 |
| `agent:{name}:cert_ready` | `agent:burger-order:cert_ready` | T1.6 | T1.11 |
| `circuit:{name}:state` | `circuit:burger-order:state` | T3.6 | T3.6, Temporal workflow pre-check |
| `circuit:{name}:stats` (hash) | `circuit:burger-order:stats` | T3.6 | T3.6 |
| `stream:workflow:{workflow_id}:context` | `stream:workflow:wf-abc123:context` | Orchestrator (T2.x) | All agents in workflow |
| `graph:{name}:{role}` (cache) | `graph:burger-order:order-agent` | T2.4 (cache on read) | T2.4 |
| `certified_kb:{name}:{query_hash}` (cache) | `certified_kb:burger-order:a1b2c3` | T2.3 (cache on read) | T2.3 |

### Key Builder Module

```python
# src/core/redis_keys.py

"""
Centralized Redis key construction. Every Redis key in the framework is built
via one of these functions — never construct keys with raw f-strings elsewhere.
This prevents typos and makes a global rename a one-file change.
"""


def agent_phase_key(agent_type: str) -> str:
    return f"agent:{agent_type}:phase"


def agent_learning_rate_key(agent_type: str) -> str:
    return f"agent:{agent_type}:learning_rate"


def agent_readiness_score_key(agent_type: str) -> str:
    return f"agent:{agent_type}:readiness_score"


def agent_proposal_ready_key(agent_type: str) -> str:
    return f"agent:{agent_type}:proposal_ready"


def agent_cert_ready_key(agent_type: str) -> str:
    return f"agent:{agent_type}:cert_ready"


def circuit_state_key(agent_type: str) -> str:
    return f"circuit:{agent_type}:state"


def circuit_stats_key(agent_type: str) -> str:
    return f"circuit:{agent_type}:stats"


def workflow_context_stream_key(workflow_id: str) -> str:
    return f"stream:workflow:{workflow_id}:context"


def graph_cache_key(agent_type: str, role: str) -> str:
    return f"graph:{agent_type}:{role}"


def certified_kb_cache_key(agent_type: str, query_hash: str) -> str:
    return f"certified_kb:{agent_type}:{query_hash}"
```

### Operational State Helper

A higher-level helper used by T1.1, T1.11, and T2.8 to read/write the full operational
state bundle atomically:

```python
# Add to src/infra/redis_client.py

from src.core.redis_keys import (
    agent_phase_key, agent_learning_rate_key, agent_readiness_score_key,
    agent_proposal_ready_key, agent_cert_ready_key,
)
from src.core.enums import Phase


def get_agent_state(agent_type: str) -> dict:
    """Read the full operational state bundle for an agent type."""
    client = get_client()
    pipe = client.pipeline()
    pipe.get(agent_phase_key(agent_type))
    pipe.get(agent_learning_rate_key(agent_type))
    pipe.get(agent_readiness_score_key(agent_type))
    pipe.get(agent_proposal_ready_key(agent_type))
    pipe.get(agent_cert_ready_key(agent_type))
    phase, lr, score, prop_ready, cert_ready = pipe.execute()

    return {
        "phase": phase or Phase.APPRENTICE.value,
        "learning_rate": float(lr) if lr else 1.0,
        "readiness_score": float(score) if score else 0.0,
        "proposal_ready": prop_ready == "true",
        "cert_ready": cert_ready == "true",
    }


def set_agent_phase(agent_type: str, phase: Phase, learning_rate: float) -> None:
    """Atomically update phase + learning_rate — used by Cert 1 and Cert 2 gates."""
    client = get_client()
    pipe = client.pipeline()
    pipe.set(agent_phase_key(agent_type), phase.value)
    pipe.set(agent_learning_rate_key(agent_type), str(learning_rate))
    pipe.execute()
```

## Acceptance Criteria

1. Every key pattern in the table above has a corresponding builder function in `redis_keys.py`
2. `get_agent_state("burger-order")` on a freshly-created agent type (after T1.1) returns
   `{"phase": "apprentice", "learning_rate": 1.0, "readiness_score": 0.0, "proposal_ready": False, "cert_ready": False}`
3. `set_agent_phase("burger-order", Phase.JOURNEYMAN, 0.5)` updates both `phase` and
   `learning_rate` atomically (verified via pipeline — both succeed or both fail)
4. `workflow_context_stream_key("wf-abc123")` returns `"stream:workflow:wf-abc123:context"` —
   matches the pattern used by T0.3's stream tests
5. No raw f-string Redis key construction exists outside `redis_keys.py` and `redis_client.py`
   (grep check)

```python
def test_no_raw_fstring_redis_keys_outside_allowed_files():
    """
    Grep check: no f-string Redis key patterns like f"agent:{...}" should appear
    outside src/core/redis_keys.py and src/infra/redis_client.py.
    """
    import pathlib
    import re

    pattern = re.compile(r'f["\'](agent|circuit|stream|graph|certified_kb):')
    allowed_files = {"redis_keys.py", "redis_client.py"}

    violations = []
    for path in pathlib.Path("src").rglob("*.py"):
        if path.name in allowed_files:
            continue
        text = path.read_text()
        if pattern.search(text):
            violations.append(str(path))

    assert violations == [], f"Raw Redis key construction found in: {violations}"
```

## Notes

- **This task is primarily a contract, not new infrastructure.** T0.3 already provides the
  Redis client. T0.8's job is to make every key name a function call, not a string literal —
  this is what lets T1.1, T1.6, and T3.6 (all written independently) agree on exactly what
  a given key looks like without copy-pasting strings between task cards.
- **Pipelines for atomicity:** `get_agent_state` and `set_agent_phase` use Redis pipelines.
  Redis pipelines are not transactions in the Postgres sense, but they guarantee the commands
  execute back-to-back without another client's commands interleaving — sufficient for our
  consistency needs here (single-writer-per-agent-type pattern).
- **Cache keys (`graph_cache_key`, `certified_kb_cache_key`) are introduced here but populated
  in T2.3/T2.4.** Defining the naming convention now means those later tasks don't need to
  invent it.
- **The grep-based test is intentionally strict.** If a future task needs a new key pattern,
  the correct fix is to add a builder function here, not to add an exception to the grep.
